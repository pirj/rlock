Next, extend the provisioning heredoc inside cmd_new. The current provisioning block (lines 93-100) is:

Replace it with an expanded version that also installs mise and configures env vars (per D-11, D-12, D-13 and Pitfall 3):
```bash
    provision_output=$(aq exec "$vm_name" <<'PROVISION'
set -e

# Base packages (Phase 1)
apk add --no-cache tmux git bash curl

# Enable community repository for mise (Pitfall 3: community repo not enabled by default in aq)
sed -i 's|^#\(.*community\)|\1|' /etc/apk/repositories
apk update

# Install mise-en-place for environment variable management (D-11)
apk add --no-cache mise

# Generate mise.toml with proxy URLs and dummy API keys (D-12, D-13)
cat > /root/mise.toml <<'MISE'
[env]
ANTHROPIC_BASE_URL = "http://10.0.2.2:9110"
OPENAI_BASE_URL = "http://10.0.2.2:9111"
ANTHROPIC_API_KEY = "dummy"
OPENAI_API_KEY = "dummy"
MISE

# Activate mise in shell profiles (ash for Alpine default, bash for agent tooling)
echo 'eval "$(mise activate sh)"' >> /root/.profile
echo 'eval "$(mise activate bash)"' >> /root/.bashrc

mkdir -p /root/repo
echo "PROVISION_OK"
PROVISION
    )
```

Key details:
- `bash` and `curl` added to base apk install (bash needed by some agent tooling per CLAUDE.md, curl for health checks)
- The sed command uncomments the community repository line in /etc/apk/repositories (Pitfall 3)
- `apk update` runs AFTER enabling community repo so the mise package index is available
- mise.toml placed at /root/mise.toml (home directory, not repo-specific)
- ANTHROPIC_BASE_URL=http://10.0.2.2:9110 and OPENAI_BASE_URL=http://10.0.2.2:9111 per D-04
- Dummy keys (ANTHROPIC_API_KEY=dummy, OPENAI_API_KEY=dummy) per D-05, D-06
- mise activation in both .profile (ash/sh) and .bashrc (bash) per research recommendation

The spinner message changes from "Installing packages" to "Provisioning guest" since it now does more than package installation.

Do NOT modify cmd_rm -- per D-09, Caddy stays running. No Caddy stop logic needed.

  <action>
Human verifies the complete security boundary works end-to-end. This checkpoint validates that:
(1) Caddy proxy starts and injects real API key headers on the host side,
(2) Guest VM has mise with proxy URLs and dummy API keys,
(3) No real API key exists inside the guest VM,
(4) Requests from guest to host proxy reach upstream APIs with correct auth.

**Pre-requisites:** Export ANTHROPIC_API_KEY in your shell (your real key).

**Steps:**

1. If you have an existing airlock, destroy it: `rl rm`

2. Create a new airlock: `rl new`
   - Expected: Spinners for "Starting API proxy", "Creating VM", "Booting VM", "Waiting for SSH", "Provisioning guest", then success.

3. Verify Caddy is running on host:
   `curl -s -o /dev/null -w "%{http_code}" http://127.0.0.1:9110`
   - Expected: Any HTTP response code (proxy is listening).

4. Connect to guest: `rl code`
   Inside tmux, run: `mise env | grep -E 'ANTHROPIC|OPENAI'`
   - Expected: ANTHROPIC_BASE_URL=http://10.0.2.2:9110, OPENAI_BASE_URL=http://10.0.2.2:9111, ANTHROPIC_API_KEY=dummy, OPENAI_API_KEY=dummy

5. Verify no real keys in guest: `env | grep -i 'api.key\|secret\|token' | grep -v dummy`
   - Expected: Empty output.

6. Test proxy from guest:
   `curl -s http://10.0.2.2:9110/v1/messages -H "Content-Type: application/json" -H "x-api-key: dummy" -H "anthropic-version: 2023-06-01" -d '{"model":"claude-sonnet-4-20250514","max_tokens":5,"messages":[{"role":"user","content":"hi"}]}' | head -c 200`
   - Expected: JSON response from Anthropic (not a connection error).

7. Exit guest (Ctrl-B D or exit).

8. Verify Caddy survives rl rm:
   `rl rm && curl -s -o /dev/null -w "%{http_code}" http://127.0.0.1:9110`
   - Expected: Caddy still responds.
  </action>
